# Deployment and Development Guide

This guide explains how to install, run, uninstall, and develop Media Hit Assistant by platform and CPU architecture.

## Verified package matrix

| Platform | Architecture | User path | Asset or command | Status |
|---|---:|---|---|---|
| Linux desktop/server with GUI libraries | amd64 / x86_64 | Normal user installation | GitHub Release `media-hit-assistant-<version>-linux-amd64.tar.gz` | Native CI package |
| Linux desktop/server with GUI libraries | arm64 / aarch64 | Raspberry Pi, RK3588, NanoPC, ARM workstation | Build from source with Qt6 packages | Source-supported and locally verified on aarch64 |
| Windows 10/11 | x64 | Normal desktop usage | GitHub Release `media-hit-assistant-<version>-windows-x64.zip` | Native CI package |
| Docker / CI | amd64 or arm64 host with Qt packages | Automated verification | `docker build` or `./scripts/verify-all.sh` | Headless self-test path |
| macOS | arm64 or x64 | Developers only | Source build after installing Qt6 manually | Not yet released as a signed asset |

## Screenshot

The repository includes a rendered preview asset, and local verification can create a real runtime screenshot.

![Dashboard preview](assets/dashboard-preview.svg)

When you need a fresh screenshot from the actual binary:

```bash
cmake --build build -j2
QT_QPA_PLATFORM=vnc QT_QUICK_BACKEND=software ./build/media-hit-assistant --screenshot docs/assets/local-install-dashboard.png
```

The command launches the same QML application, loads sample articles, captures the first dashboard window, saves a PNG, and exits. It is useful on CI or servers without a physical display.

## Linux amd64 deployment from Release archive

### What it is

Use this path for normal Linux x86_64 desktops, cloud desktops, and CI runners. The Release archive is an installed-tree package produced by GitHub Actions on Ubuntu 24.04.

### Install

```bash
VERSION=0.1.1
curl -L -o media-hit-assistant-linux-amd64.tar.gz \
  "https://github.com/lxy798132784/Self-media-Viral-Assistant-Jizhile/releases/download/v${VERSION}/media-hit-assistant-${VERSION}-linux-amd64.tar.gz"
tar -xzf media-hit-assistant-linux-amd64.tar.gz
cd "media-hit-assistant-${VERSION}-linux-amd64"
./usr/bin/media-hit-assistant
```

### Verify before running

```bash
curl -L -o SHA256SUMS \
  "https://github.com/lxy798132784/Self-media-Viral-Assistant-Jizhile/releases/download/v${VERSION}/SHA256SUMS"
sha256sum media-hit-assistant-linux-amd64.tar.gz
grep "linux-amd64" SHA256SUMS
```

### Uninstall

If you used the extracted archive only, remove the extracted folder and downloaded file:

```bash
cd ..
rm -rf "media-hit-assistant-${VERSION}-linux-amd64" media-hit-assistant-linux-amd64.tar.gz SHA256SUMS
```

If you installed it into a prefix with CMake, remove that prefix:

```bash
rm -rf /tmp/media-hit-install
```

## Linux arm64 / aarch64 deployment from source

### What it is

Use this path for Raspberry Pi, RK3588 boards, NanoPC-T6, ARM servers, and other aarch64 machines. The source tree builds the native binary on the device, so no cross-compiler is required.

### Dependencies on Ubuntu 22.04 / 24.04

```bash
sudo apt-get update
sudo apt-get install -y cmake g++ python3 \
  qt6-base-dev qt6-declarative-dev qt6-tools-dev qt6-tools-dev-tools \
  qml6-module-qtquick qml6-module-qtquick-controls qml6-module-qtquick-layouts
```

### Build, test, and run

```bash
git clone https://github.com/lxy798132784/Self-media-Viral-Assistant-Jizhile.git
cd Self-media-Viral-Assistant-Jizhile
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j2
ctest --test-dir build --output-on-failure
QT_QPA_PLATFORM=offscreen ./build/media-hit-assistant --self-test
./build/media-hit-assistant
```

### Local install into an isolated prefix

```bash
cmake --install build --prefix /tmp/media-hit-install
/tmp/media-hit-install/bin/media-hit-assistant
```

This keeps system directories untouched. It installs the executable, README, changelog, docs, icon, desktop metadata, plugin directory, and bundled data service index under `/tmp/media-hit-install`.

### Uninstall

```bash
rm -rf /tmp/media-hit-install
```

## Windows x64 deployment

### What it is

Use this path for Windows 10/11 x64. The zip is built on `windows-2022` in GitHub Actions and includes the application folder generated by the Windows packaging script.

### Install and run

1. Download `media-hit-assistant-<version>-windows-x64.zip` from the GitHub Release page.
2. Right-click the zip file and choose **Extract All**.
3. Open the extracted folder.
4. Double-click `media-hit-assistant.exe`.

PowerShell equivalent:

```powershell
$Version = "0.1.1"
Invoke-WebRequest `
  -Uri "https://github.com/lxy798132784/Self-media-Viral-Assistant-Jizhile/releases/download/v$Version/media-hit-assistant-$Version-windows-x64.zip" `
  -OutFile "media-hit-assistant-windows-x64.zip"
Expand-Archive .\media-hit-assistant-windows-x64.zip -DestinationPath .\media-hit-assistant
.\media-hit-assistant\media-hit-assistant.exe
```

### If the window closes immediately

If double-clicking `media-hit-assistant.exe` opens a command window that disappears immediately, run the diagnostic launcher from the same folder instead:

```powershell
.\media-hit-assistant\run-with-log.bat
```

It writes `media-hit-assistant.log` next to the executable and keeps the console open when startup fails. This makes missing Qt/runtime DLLs, unsigned-app security blocking, and QML/graphics/path startup failures visible.

New Windows packages run `media-hit-assistant.exe --self-test` from the packaged distribution directory and must include `run-with-log.bat`, so the Release gate checks the same layout that users extract.

### Uninstall

Delete the extracted `media-hit-assistant` folder and the downloaded zip. The app does not require a Windows installer service.

## Windows x64 development build

### Dependencies

Install:

- Visual Studio 2022 with **Desktop development with C++**;
- CMake;
- Qt 6.7.x for MSVC x64;
- Python 3.

### Build and package

Open “x64 Native Tools Command Prompt for VS 2022” or a PowerShell session with the MSVC environment loaded:

```powershell
.\scripts\package-windows.ps1 -BuildDir build-win -Config Release -DistDir media-hit-assistant-dev-windows-x64
```

The script configures CMake, builds the binary, runs CTest, runs the offscreen self-test, calls `windeployqt` when available, copies docs/plugins/vendor assets, and writes a zip package.

## Docker and CI usage

### Headless verification

```bash
./scripts/verify-all.sh
```

The full gate runs:

1. CMake build;
2. CTest unit tests;
3. offscreen self-test;
4. Markdown/XML artifact checks;
5. QML control audit;
6. documentation alignment audit;
7. offscreen launch smoke.

### Docker

```bash
docker build -t media-hit-assistant .
docker run --rm media-hit-assistant
```

Docker is intended for verification and automation, not as the primary end-user GUI delivery method.

## Architecture-specific notes

### amd64 / x86_64

This is the primary prebuilt Linux target. Prefer the Release archive when you only want to use the app. Build from source when changing C++ or QML.

### arm64 / aarch64

Build locally from source. Qt6 is available through Ubuntu packages on many ARM boards, and the project has no architecture-specific C++ code. Use `-j2` on small boards to avoid memory pressure.

### Windows x64

Use the prebuilt zip for normal users. Use `scripts/package-windows.ps1` for development and release reproduction.

### macOS

There is no signed macOS package yet. Developers can build manually with Qt6 and CMake, but Release assets currently cover Linux amd64 and Windows x64.

## Development workflow by role

### End user

1. Download the matching Release asset.
2. Verify checksum if needed.
3. Extract.
4. Run the executable.
5. Keep credentials only in local settings.

### Developer

1. Install Qt6 and CMake.
2. Build with `cmake -S . -B build -DCMAKE_BUILD_TYPE=Release`.
3. Run `ctest --test-dir build --output-on-failure`.
4. Run `./scripts/verify-all.sh` before committing.
5. Update docs and screenshots when visible behavior changes.

### Release maintainer

1. Push a `v*` tag.
2. GitHub Actions builds Linux amd64 and Windows x64 assets.
3. The publish job uploads source archives, docs bundle, Release notes, and `SHA256SUMS`.
4. Download at least one asset and verify structure and checksum.

## Runtime data and credentials

Local tests do not need credentials. If no access key is configured, the app uses safe sample collection. Real access keys must stay in local settings or environment variables and must not be committed to the repository.
